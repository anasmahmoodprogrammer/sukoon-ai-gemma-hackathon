1. Introduction
1.1 Purpose
This document specifies the functional and non-functional requirements for Sukoon AI, an Urdu-first conversational well-being companion built on Gemma 4 for the Build with Gemma NUTECH Islamabad hackathon. It is written to be implementable within the hackathon's compressed timeline and to map directly onto the published judging criteria.
1.2 Scope
Sukoon AI is a web-based chat companion that listens, validates feelings, and offers culturally appropriate coping suggestions in Urdu script, Roman Urdu, or English. It is explicitly not a diagnostic or clinical tool. In scope for the hackathon build: text chat, an optional voice input, a safety/crisis-escalation layer, and an anonymized analytics view for the demo. Out of scope for the hackathon build: user accounts, payment, native mobile apps, and integration with electronic health records.
1.3 Intended audience
Primary users: Pakistani residents experiencing everyday stress, anxiety, or isolation who are unlikely to seek formal help due to stigma.
At-risk users: people expressing self-harm or crisis-level distress, who need to be routed to real human help, not kept in conversation with the AI.
Secondary audience: hackathon judges evaluating Gemma integration, innovation, functionality, and the writeup.
User characteristics vary widely in English literacy, smartphone comfort, and willingness to discuss mental health at all — the design should not assume a tech-fluent, English-fluent user.

2. Overall description
2.1 Product perspective
Sukoon AI is a new, standalone product. It positions itself alongside — never as a replacement for — existing licensed mental-health services and crisis helplines in Pakistan (Appendix B). Its value is being the low-friction first step: anonymous, judgment-free, and in the user's own language, for people who would not otherwise reach out at all.
2.2 Product functions (summary)
Culturally aware, Urdu-first conversational support (Module A).
Optional voice input using Gemma 4's native audio understanding (Module B).
Real-time safety classification and crisis escalation (Module C).
Privacy-first, anonymous-by-default data handling (Module D).
Accessible, low-bandwidth-friendly interface (Module E).
Anonymized impact dashboard for the demo (Module F).
Optional de-stigmatization / awareness features (Module G).

2.4 Operating environment
Modern evergreen mobile or desktop browser; Node.js backend; Gemma 4 accessed via API or local/edge inference for the smaller model sizes. No native app required for the hackathon build.
2.5 Design and implementation constraints
Per the hackathon rules, no code may be pushed to GitHub and no repository created before hackathon day (see Phase 0/1).
The Kaggle Writeup and final submission are due July 15, alongside the live presentation at the NUTECH Seminar Hall.
The core model must be Gemma 4, per the hackathon's judging rubric.
Build time is effectively a single hackathon day — the phased plan in Section 8 is time-boxed accordingly.
2.6 Assumptions and dependencies
Gemma 4 API access/quota is available and stable through the demo window.
The venue provides reliable internet for a live, unscripted demo.
Listed helpline numbers remain reachable — verify immediately before the demo (Appendix B).
Users have access to a smartphone or computer with a modern browser.

5. External interface requirements
5.1 User interfaces
Single-window chat view, mic button, and a persistent helpline access point (FR C4).
Dedicated, visually distinct safety card for crisis-level replies (FR C2).
Mixed-direction text rendering for Urdu script and Roman Urdu/English (FR E1).
5.2 Hardware interfaces
Device microphone, accessed via standard browser permissions, for voice input (Module B).
5.3 Software interfaces
Gemma 4 API (or local/edge inference for E2B/E4B sizes) for conversation and risk classification.
A lightweight datastore (SQLite for the demo build is sufficient) for anonymized aggregate trend data only.
5.4 Communication interfaces
HTTPS for all client-server traffic; TLS 1.2 or later (NFR-S security requirements).
WebSocket or server-sent events for streaming replies (FR A3).
6. System architecture overview
At a high level, a user message (voice or text) is sent to the Gemma 4 conversational core, which replies using the Sukoon persona system prompt. In parallel, a structured risk-classification pass scores the same turn. Low or moderate risk produces a comforting, culturally aware reply; crisis-level risk produces a brief, calm reply plus an immediate safety card with real helpline numbers. Both paths write only anonymized, aggregate sentiment data to storage — never raw message content tied to an identifier. This flow was illustrated as a diagram earlier in this conversation and should be reproduced in the Kaggle Writeup.
7. Safety requirements (non-negotiable)
Because Sukoon AI will be used by people in real emotional distress, the following are treated as release-blocking, not aspirational. They consolidate Module C and the ethical NFRs above.
Must never
Provide a clinical diagnosis or a medication/dosage recommendation, in any language or framing.
Simulate being a licensed therapist or claim clinical credentials.
Bury or hide helpline access behind menus during a crisis-flagged conversation.
Store raw, crisis-flagged conversation content.
Must always
Complete risk classification before a reply is shown to the user (FR C1).
Surface a real, verified Pakistani helpline within 2 seconds of detecting crisis-level risk (NFR-E2).
Keep the disclaimer and helpline access visible at all times, not only when risk is detected (FR C4, C6).

Appendix A — sample Gemma 4 system prompt
Condensed prompt operationalizing FR A2, C1, C3, and C6. Expand with a few worked examples during Phase 2, and keep it in a config file separate from application code (NFR-M1).
You are Sukoon, a warm, non-judgmental listener for people in Pakistan.Always reply in the user's own register: Urdu script, Roman Urdu, or English.Validate feelings first. Offer gentle, practical coping ideas suited to local life. Never give a diagnosis, never give a medication or dosage, never claim to be a licensed therapist.Respect the user's chosen framing (secular or faith-informed) for coping suggestions.On every turn, in addition to your reply, also output a JSON risk object: {"risk_level": "none|elevated|crisis", "flags": [...]}.If risk_level is "crisis": keep your reply brief and calm. Do not try to counsel the crisis away yourself — the application will surface the safety card. Your job is to stay warm, not to resolve the emergency.
